Scanner-First MVP

Single Card Identity Flow

This artifact intentionally ignores pricing, bundle totals, eBay sync, and other dealer workflows. The only question it tries to answer is whether a dealer can scan one card photo and get a trustworthy match.

  • One photo in
  • One likely card match out
  • Confidence shown honestly
  • Manual recovery when the scan is wrong
What To Evaluate

Trust and recovery, not polish

  • Does the scanner UI make it obvious that this is one card at a time?
  • Does the result screen show the right identity fields?
  • When the model is wrong, is correction fast and clear?
  • Is confidence visible enough to prevent false trust?
Goal

Prove Identification Works

Before price data, bundle math, or deal logging, the app needs a reliable identity layer.

Scope

Only 3 Screens

Scanner, Match Result, and Match Alternatives / Search. Anything else is a later phase.

Mode

Convention-Floor Constraints

Dark mode, large tap targets, minimal typing, and graceful handling when the first guess is wrong.

Open Questions

What We Need To Decide Before Building

Camera

Capture vs Import Default

If the app is used live on the floor, camera should be primary. If testing starts with saved photos, import may be the first implemented path.

Confidence

How Honest Should The UI Be?

It should probably be very explicit. A wrong confident result is more damaging than a cautious low-confidence one.

Search

Manual Search Scope

For the first version, search should probably support card name, set name, and card number only.

Next Layer

Pricing Comes After Identity

Once this flow works, the next screen addition should be pricing on the confirmed match, not bundle logic yet.
